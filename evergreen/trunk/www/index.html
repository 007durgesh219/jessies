Edit

Why another editor?

You can be forgiven for wondering why anyone would write another editor. There are plenty out there. Surely one fits?

Edit started as a project to reimplement Rob Pike's acme editor in Java, so I could use it on Unix and Windows. (I'd got used to it at university, using Plan 9.) Eventually, my editor diverged and today probably represents my personality quite closely; in particular, if the computer can do something, it must. I certainly shouldn't have to.

What's so great about Edit?

Ubiquitous regular expressions

Programmers know regular expressions. And yet their graphical tools don't use make much use of them. Edit does. Anywhere it's asking you to type something, it's probably expecting a regular expression: full Perl/Java regular expressions in find, in find/replace, in find in files (as you'd expect), but also in the open dialog.

The best open dialog

Edit indexes your project, so you don't have to remember where files are. Or what case your co-worker used ("Hyperlink" or "HyperLink"?). Just a substring or regular expression that matches what you want. Edit updates a list of the matching files as you type (like iTunes):

The best find

You have a fast computer, yet the 'find' function in other editors is grindingly slow. Edit, like less(1), highlights all the matches. Only Edit does it as you type, lets you move backwards as easily as forwards through the matches, and shows you next to the scroll bar just where the matches are:

The best find/replace

Find/replace functionality in other editors has a habit of being awkward in use. How much better to be able to see all the changes at once, with the results of the substitutions?

(An open question is whether we wouldn't be even more comfortable with something more like the output from diff -u.)

The best spelling checking

The best ctags support

Edit uses Exuberant ctags (if installed) to understand the structure of your file. You can click on a tag to go to that point in the file, and the tag corresponding to the current caret position is always highlighted.

Classes, methods and fields are easily distinguished by shape; accessibility is denoted by color. Static members are in bold.

How do I install it?

Good question. The distribution is here (edit.tgz), but you'll also need a copy of the library to go with it (salma-hayek.tgz).

The ChangeLog is available separately.

If you have access to my home directory, run sudo ~elliotth/Projects/salma-hayek/install-everything.sh, and make sure that /usr/local/bin is high up your path.

Notes on using Edit

Workspaces are important. You probably want one for each project you're working on. If a workspace looks like a project (i.e. has a Makefile or is under revision control), the files will be indexed. So you probably want the root of your workspaces to be the directories with the top-level makefiles.

Learn to live with Edit's idea of code formatting. If you find this hard to swallow, read Ken Arnold's thoughts on coding style first. What you gain from accepting Edit's style (which shouldn't be a problem for Java programmers, because it's pretty much expecting that in all the C family) is that Edit will do the formatting for you. Even correcting stuff you paste in.
